PIPELINE. Документ фиксирует сквозной pipeline системы Tirehub ETL. Полный pipeline: supplier email or supplier file -> connector intake -> evidence -> landing -> supplier parser -> NDJSON emit -> schema validation -> rulesets -> canonical storage -> curated layer -> serving layer -> PostgreSQL. CONNECTOR STAGE. Connector получает письмо или supplier file. Для mail intake текущий вход: IMAP mailbox. Connector определяет новые письма по UID watermark. EVIDENCE STAGE. Каждое обработанное письмо сохраняется как evidence artifact. Это обеспечивает auditability и replay capability. LANDING STAGE. Каждое вложение сначала попадает в landing layer. LANDING хранит сырой attachment до routing и последующего ingestion. ROUTING STAGE. Attachment маршрутизируется в inputs/inbox/<supplier>. ROUTING использует supplier registry. PARSER STAGE. Supplier-specific parser читает routed file и преобразует его в NDJSON через emitter. VALIDATION STAGE. NDJSON проходит schema validation и contract checks. RULESET STAGE. NDJSON проходит sanity, normalization и impact rules. CANONICAL STORAGE STAGE. Валидные факты записываются в canonical storage или SSOT layer. CURATED STAGE. Canonical facts преобразуются в curated output. SERVING STAGE. Curated output публикуется в serving schema и PostgreSQL. CURRENT VERIFIED SEGMENT. На текущий момент verified segment pipeline: IMAP -> mail_ingest_worker -> evidence -> landing -> supplier routing -> watermark update. NEXT VERIFIED SEGMENT. Следующий обязательный шаг: routed supplier file -> parser -> NDJSON -> schema validation -> canonical storage.
