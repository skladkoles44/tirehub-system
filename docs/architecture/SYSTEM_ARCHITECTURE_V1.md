SYSTEM ARCHITECTURE V1. Документ фиксирует фактические архитектурные слои системы Tirehub ETL. Система является supplier data ingestion platform для входящих файлов поставщиков автотоваров. Основные слои системы: connectors; ingestion; contracts; schemas; rulesets; curated; serving; sql storage. CONNECTORS LAYER. Слой connectors отвечает за внешние входы в систему. Текущий реализованный connector: scripts/connectors/mail_ingest_worker_v1.py. Connector получает письма по IMAP, сохраняет evidence, кладет вложения в landing и маршрутизирует их в supplier inbox. INGESTION LAYER. Слой ingestion отвечает за преобразование входящих supplier files в NDJSON и canonical facts. Основные скрипты расположены в scripts/ingestion. Для каждого поставщика существует свой parser и emitter. CONTRACTS LAYER. Слой contracts фиксирует машинные контракты ingestion и parser-процессов. Документы расположены в docs/contracts. SCHEMAS LAYER. Слой schemas задает допустимые структуры canonical_item и rejected_item. JSON schema файлы расположены в schemas. RULESETS LAYER. Слой rulesets отвечает за data quality, sanity checks, signature normalization, token field extraction и impact weights. CURATED LAYER. Слой curated отвечает за подготовку производных нормализованных наборов данных для дальнейшего использования. Основной скрипт: scripts/curated/tirehub_curate_v1.py. SERVING LAYER. Слой serving отвечает за выкладку подготовленных данных в serving schema и PostgreSQL. Основной скрипт: scripts/serving/apply_to_postgres_v1.py. SQL STORAGE LAYER. Слой sql содержит schema definitions, migrations и serving schema. Основные каталоги: sql/migrations и sql/serving. ARCHITECTURE RULE. Система должна сохранять разделение repo; runtime; data; external inputs. Репозиторий не должен хранить runtime state и бинарные runtime artifacts. CURRENT STATUS. На текущий момент слой connectors и ingestion подтвержден реальным mail intake кейсом. Слои contracts, schemas, rulesets, curated, serving и sql присутствуют в репозитории и требуют синхронизации с PROJECT MAP и operational docs.
